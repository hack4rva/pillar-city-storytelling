
# RVA Arts Events, Fast: A Practical Map to RSS and iCal Sources

## Executive Summary

For hackathon teams looking to ingest Richmond arts events quickly, iCal (ICS) endpoints are vastly superior to RSS feeds. The most reliable structured data comes from organizations using WordPress with "The Events Calendar" plugin, which natively exposes ICS exports. 

Key findings from the current data sweep include:
* **ICS is the gold standard for events:** Richmond Symphony and Style Weekly offer robust ICS endpoints that include critical fields like start/end times, descriptions, and locations [1] [2].
* **Style Weekly acts as a vital aggregator:** While Style Weekly maintains a general RSS feed for its editorial content [3], its true value lies in its event-specific ICS exports, including venue-specific feeds that can backfill data for organizations lacking their own structured feeds (like the Visual Arts Center of Richmond) [4].
* **Direct organizational feeds are inconsistent:** Custom-built calendars, such as the one used by Virginia Repertory Theatre, often lack visible ICS or RSS feeds [5]. 
* **Recurrence is a major risk:** Event feeds often include recurring rules (RRULE) or span massive historical archives (e.g., back to 2015) [1]. Teams must implement strict time-bounding and recurrence expansion to avoid database bloat.

## Parsing Stack Recommendations

To process these feeds efficiently in a hackathon, select a mature parsing stack that handles recurrence and timezones natively.

| Language | Library | Primary Use | Strengths | Watch-outs |
| :--- | :--- | :--- | :--- | :--- |
| **Python** | `icalendar` + `recurring-ical-events` | Parse ICS & expand `RRULE` | Stable, well-documented; excellent recurrence support. | Supplement with `zoneinfo` for edge-case timezones. |
| **Python** | `feedparser` | Parse RSS/Atom | Handles malformed XML gracefully. | Not event-specific; requires custom field mapping. |
| **Node.js** | `node-ical` | Parse ICS | Fast and widely adopted. | Must be combined with an expansion library. |
| **Node.js** | `ical-expander` | `RRULE` expansion | Handles recurring events cleanly. | Ensure timezone-aware expansion using Luxon. |
| **Node.js** | `rss-parser` | Parse RSS/Atom | Simple, promise-based API. | Limited event semantics. |

**Takeaway:** If your team is JS-heavy, use `node-ical` + `ical-expander` + `luxon`. If Python-heavy, use `icalendar` + `recurring-ical-events` + `zoneinfo`.

## Risk and Edge Cases

Prevent silent data failures by addressing these common feed quirks:

* **Recurrence Bloat:** Always expand `RRULE` and immediately filter for future instances. Respect parameters like `hide_subsequent_recurrences=1` when provided by the source [2].
* **Archive Flooding:** Unfiltered category feeds can return events dating back years (e.g., Richmond Symphony archives back to 2015) [1]. Enforce strict `DTSTART >= now` filters post-parse.
* **Protocol Quirks:** Links often surface as `webcal://` or Outlook subscription templates [1] [2]. Always normalize by replacing `webcal://` with `https://` before passing to your HTTP client.
