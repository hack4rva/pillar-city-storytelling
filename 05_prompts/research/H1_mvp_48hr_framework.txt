Section H — MVP Feasibility
Title: 48-Hour Feasibility Framework for Arts and Storytelling Tools

The most common failure mode in civic hackathon projects is misreading what "done" means. A team that spends Friday evening designing a comprehensive database schema and Saturday building authentication ends up on Sunday with a login page and no data. For City Storytelling problems specifically, the trap is conceptual richness: both arts discovery and civic storytelling are emotionally resonant themes that invite ambitious visions — a living map of Richmond's cultural soul, an AI system that translates resident stories into policy briefs — while the actual buildable unit in 48 hours is far more modest. Teams need a forcing function: on Sunday at 2pm, what will a judge see on the screen, and what will it show them that they could not see before? Every design decision must be evaluated against that single question.

For arts event discovery, the 48-hour feasibility question is whether the team can produce a working interface showing real Richmond arts events, pulled from at least one live data source, filterable in at least one useful dimension. The Eventbrite public search API (no auth required for basic queries) can return events in Richmond, VA filtered by category, date, and keyword within a few hours of a team starting. The realistic demo is: user opens tool, selects "this weekend" and "visual arts," sees 5-10 real events with venue, date, and link back to original source. That is achievable in 48 hours. What is not achievable: comprehensive coverage, automated deduplication, neighborhood-level filtering tied to actual Richmond geocoding, mobile-optimized UI with accessibility features, and any form of user account or preference system. Teams that try to build comprehensive coverage will have nothing to show; teams that build the thin, real-data slice will have a compelling demo.

For story collection, the 48-hour split is more fundamental. Story collection (building a submission interface) and story insight (extracting themes from stories) are different products requiring different architectures. The submission interface is achievable in 48 hours: a form with a few fields (neighborhood, time period, story text, optional anonymization toggle), a backend that stores submissions, and a basic display of submitted stories. The insight dashboard — showing AI-extracted themes, connections across stories, policy-relevant patterns — is a much harder problem and requires a seed dataset to demonstrate at all. Teams must choose one primary value proposition to demo. If they choose insight, they need to pre-populate the tool with seed stories before the event, because no real stories will accumulate during the hackathon. Research suggests that teams who reach out to 5-10 community members before the hackathon for real pilot story submissions produce more credible demos than teams who write fictional stories themselves.

Output
- Facts (with URLs): Document Eventbrite's public API endpoint for event search by location (docs.eventbrite.com); document what category filters exist for arts/cultural events in the Eventbrite API; identify 3-5 Richmond-specific Eventbrite organizers whose events would appear in a live search demo.
- Inferences (clearly labeled): [Inference] A team of 3-4 people that starts with Eventbrite API on Friday evening can have a working event display with real data by Saturday afternoon, leaving time for UI polish and demo prep. [Inference] A story insight tool demoed without at least 10 seed stories will not convincingly demonstrate the insight value proposition to judges.
- Unknowns (what cannot be verified publicly): Whether Eventbrite API rate limits or category structures have changed recently enough to affect 48-hour build estimates; whether any Richmond arts organizations have machine-readable event feeds beyond Eventbrite.
- Produce a feasibility scorecard for each of the two problem statements, rating the following dimensions on a 1-5 scale: data accessibility, technical complexity, demo clarity, community impact legibility, and 48-hour achievability. Include one "scope trap to avoid" for each.
