Section H — MVP Feasibility
Title: Feasibility of Building a Story Collection and Insight Tool in 48 Hours

A resident story collection tool has two distinct value propositions that require separate evaluation for 48-hour feasibility. The first is the collection interface — the experience of submitting a story — which is technically straightforward but has significant ethical requirements. A basic form with fields for neighborhood, time period (when did you live there, when did this happen), story text, optional name or anonymization choice, and a single-click submission can be built in 3-4 hours. What takes more time is building it responsibly: adding explicit consent language, a clear explanation of data use, an anonymization option, and a withdrawal contact mechanism. Teams that skip these elements to save time are building something that should not be demoed publicly, because inviting real resident story submissions without adequate consent is an ethical violation. Research what the minimum viable consent UI looks like for a civic story prototype.

The second value proposition is the insight dashboard — showing City staff or planners what themes and patterns emerge from a collection of resident stories. This is where the interesting technology lives (NLP, theme extraction, geographic clustering), but it requires a corpus of stories that does not exist at the start of a hackathon. This is the central feasibility trap: teams spend 80% of their time building sophisticated insight infrastructure and then have nothing to run it on. The solution is pre-seeding. Teams that reach out to 5-10 community members before the hackathon — through a civic organization, a neighborhood association, a cultural institution like the Valentine — and collect real pilot stories can run their insight tools on real data. Alternatively, teams can use a curated set of public oral history excerpts (the Valentine's existing digital collections include transcripts) as a demonstration dataset, with clear labeling that these are archival stories rather than new submissions.

The AI/NLP question deserves specific feasibility research. GPT API calls can extract themes from short story texts in seconds — a team can write a prompt like "Extract the three most prominent themes from this resident story, using 3-5 word phrases" and get useful output within hours of API access. However, theme extraction from a small corpus (10-20 stories) is more an illustration than genuine pattern detection, and teams must be honest about this in their demo. Clustering algorithms (LDA topic modeling, k-means on embeddings) require larger corpora to produce meaningful results and have a steep setup cost in a hackathon context. The most honest and achievable path is: use GPT API for per-story tagging, build a simple aggregation view showing which tags appear most frequently, and frame the insight demo as "imagine 200 stories ran through this system" rather than claiming the demo corpus represents community-wide insight. Research what the realistic output quality is for GPT-based theme extraction on 15-20 short personal narrative texts.
