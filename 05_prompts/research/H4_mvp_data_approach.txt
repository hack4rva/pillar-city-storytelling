Section H — MVP Feasibility
Title: Working With Fragmented Arts Data and No Single API

The defining data challenge for the arts discovery problem is that no single authoritative Richmond arts event source exists. This is not a solvable problem within a hackathon — it is the problem the tool is trying to address. Teams that spend significant time searching for "the right API" or hoping a comprehensive Richmond arts database exists are delaying the moment they need to accept: the data is fragmented, and the MVP is about demonstrating that useful aggregation is possible from imperfect, heterogeneous sources. The strategic question is which combination of sources produces enough real Richmond event data to make a compelling demo while staying within the build capacity of a 3-4 person team over 48 hours.

The recommended strategy is a layered spine approach: use Eventbrite API as the primary automated data source (high volume, structured JSON, reliable access), add 3-5 manually curated iCal feeds from anchor Richmond cultural institutions (VMFA, ICA at VCU, Black History Museum, one neighborhood arts organization), and add a small hand-seeded dataset of 5-10 events from Richmond arts organizations that have no machine-readable feed but are important for coverage legitimacy (a neighborhood arts nonprofit, a Black-owned gallery, a community theater). This combination takes roughly 6-8 hours of development for the data pipeline and produces a demo dataset rich enough to show meaningful filtering. The key discipline is not to expand coverage beyond this scope — every additional data source adds normalization complexity and debugging time that the team does not have.

Data cleaning challenges in arts event aggregation are predictable and should be researched before the hackathon starts. The most common issues: (1) Duplicate events — the same event appearing on Eventbrite and a venue's iCal feed, requiring deduplication logic based on event name, date, and venue; (2) Inconsistent venue names — "The Valentine" vs. "The Valentine Museum" vs. "1015 Clay St" appearing as the same venue in different sources; (3) Missing or imprecise geocoding — events listed with a neighborhood name but no street address, or addresses not recognized by geocoding APIs; (4) Cancelled events not removed — Eventbrite listings for cancelled events may remain publicly visible with altered descriptions but no API flag. For a 48-hour prototype, the practical approach is to not attempt automated deduplication (display the source URL prominently so users can see the original listing) and to accept that geocoding will be incomplete. The demo should show the best-case data, not claim comprehensive coverage.

For story collection tools, the seed dataset question is equally strategic. Teams must decide between three approaches before the hackathon begins: (A) fictional/composite stories written by the team to illustrate the concept (fastest to produce, lowest credibility, highest risk of being seen as inauthentic); (B) public archival stories from the Valentine Museum's oral history collection or similar public domain sources (slower to identify and curate, higher credibility, requires checking licensing); or (C) real pilot stories collected pre-hackathon from community members through a brief outreach effort (hardest logistically, most credible demo, most replicable as a real product). Research suggests that approach C, even with only 5-8 stories, produces dramatically more compelling demos than A or B, because judges can ask where the stories came from and get a real answer.
